API ENDPOINTS - SIMPLE OVERVIEW (VERIFIED AGAINST SOURCE CODE)
================================================================
Base URL: /api
Auth: Send token in header -> Authorization: Bearer <token>

--------------------------------------------------
USER ENDPOINTS (/api/user)
--------------------------------------------------

1) POST /api/user/register
   Purpose: Creates a brand new user account.

   req.body =
   {
     email: string,
     phone: string,
     password: string,
     fullName: string,   // must contain at least a first and last name (split by space)
     birthday: string
   }

   res =
   {
     success: boolean,
     message: string
   }

   Note: if the server's 18+ age requirement is enabled, registration is rejected for
   anyone under 18 based on the supplied birthday.

2) POST /api/user/login
   Purpose: Logs a user in and gives them a token to use for future requests.

   req.body =
   {
     email_or_phone: string,
     password: string
   }

   res =
   {
     success: boolean,
     message: string,
     token: string
   }

   Note: login is rejected if the account is currently banned.

3) POST /api/user/logout
   Purpose: Logs the user out by cancelling their current token.

   req.headers =
   {
     Authorization: "Bearer <token>"
   }

   req.body = {}   (nothing needed)

   res =
   {
     success: boolean,
     message: string
   }

   Note: this route reads and validates the token itself directly from the header - it
   does not go through the shared token-verification step used by other protected routes.

4) GET /api/user/me
   Purpose: Gets the profile info of the user who is currently logged in.

   req.headers =
   {
     Authorization: "Bearer <token>"
   }

   res =
   {
     success: boolean,
     message: string,
     user:
     {
       id: string,
       email: string,
       phone: string,
       fullName: string,
       birthday: string,
       createdAt: string,
       permission_level: number
     }
   }

5) PUT /api/user/me
   Purpose: Lets the logged-in user update their own account details.

   req.headers =
   {
     Authorization: "Bearer <token>"
   }

   req.body =
   {
     password?: string,
     email?: string,
     phone?: string,
     fullName?: string,
     birthday?: string
   }

   res =
   {
     success: boolean,
     message: string,
     user?:
     {
       id: string,
       email: string,
       phone: string,
       fullName: string,
       birthday: string,
       createdAt: string,
       permission_level: number
     }
   }

   Note: at least one field must actually change, otherwise the request is rejected with
   "No changes to update". A duplicate email/phone is rejected with a clear message
   rather than a raw database error.

PROFILE ENDPOINTS (/api/profile)
--------------------------------------------------
(All profile endpoints need: req.headers = { Authorization: "Bearer <token>" })
(A "profile" is like a Netflix-style profile - one user account can have several profiles,
up to a server-configured maximum.)

6) POST /api/profile/
   Purpose: Creates a new profile under the logged-in user's account (fails once the
   server-configured maximum profile count is reached).

   req.body = {}   (nothing needed)

   res =
   {
     success: boolean,
     message: string,
     profiles:
     [
       { id: string, profileName: string, age: number, ImageName: string },
       ...
     ]
   }

7) PUT /api/profile/
   Purpose: Updates several profiles belonging to the user in one request.

   req.body =
   {
     updates:
     [
       { profileId: string, profileName?: string, age?: number, ImageName?: string },
       ...
     ]
   }

   res =
   {
     success: boolean,
     message: string,
     profiles:
     [
       { id: string, profileName: string, age: number, ImageName: string },
       ...
     ]
   }

   Note: any profileId in the updates array that does not belong to the logged-in user
   is silently skipped (no error) rather than rejecting the whole request.

12) GET /api/profile/:profileId/details
    Purpose: Gets the complete details of one profile, including watch history and liked content.

    req.params =
    {
      profileId: string
    }

    res =
    {
      success: boolean,
      profile:
      {
        id: string,
        profileName: string,
        age: number,
        ImageName: string,
        likedContentIds: [ contentId, ... ],
        lastWatched:
        [
          { episode_id: string, content_id: string, position_seconds: number },
          ...
        ]
      }
    }

    Note: lastWatched has one entry per content the profile has ever watched (not one per
    episode) - each entry holds the specific episode last watched within that content,
    plus how many seconds into that episode the profile last stopped at (position_seconds),
    and the list is capped at a server-configured maximum length.

13) POST /api/profile/:profileId/likes/:contentId
    Purpose: Adds or removes a "like" on a specific movie/show for a specific profile (toggle).

    req.params =
    {
      profileId: string,
      contentId: string
    }

    req.body = {}   (nothing needed)

    res =
    {
      success: boolean,
      message: string,
      liked: boolean,
      likedContentIds: [ contentId, ... ]
    }

    Note: if the server's age-limit check is enabled, this is rejected when the profile's
    age is below the content's age_limit.

14) POST /api/profile/:profileId/watch/:contentId
    Purpose: Records that a profile just watched a movie/show, with no specific episode given.
    Behavior: if this content already has a saved entry in the profile's watch history, resumes
    from that saved episode; otherwise starts from season 1, episode 1. Fails with a clear
    message if the content has no episodes at all yet.

    req.params =
    {
      profileId: string,
      contentId: string
    }

    req.body = {}   (nothing needed)

    res =
    {
      success: boolean,
      message: string,
      episode:
      {
        id: string,
        contentId: string,
        seasonNumber: number,
        episodeNumber: number,
        title: string,
        videoUrl: string
      },
      lastWatched:
      [
        { episode_id: string, content_id: string, position_seconds: number },
        ...
      ]
    }

    Note: if the server's age-limit check is enabled, this is rejected when the profile's
    age is below the content's age_limit.
    Note: position_seconds on the resulting entry (lastWatched[0]) is only carried over
    from before when resuming the SAME episode that was already tracked for this content -
    switching to a different episode, or having no prior entry, always starts it at 0. Use
    endpoint 16 below to update position_seconds frequently during playback; this endpoint
    is only meant to be called once per "started watching this" event, not on every tick.

15) POST /api/profile/:profileId/watch/:contentId/:episodeId
    Purpose: Same as above, but records a specific episode as watched instead of resuming/defaulting.

    req.params =
    {
      profileId: string,
      contentId: string,
      episodeId: string
    }

    req.body = {}   (nothing needed)

    res =
    {
      success: boolean,
      message: string,
      episode:
      {
        id: string,
        contentId: string,
        seasonNumber: number,
        episodeNumber: number,
        title: string,
        videoUrl: string
      },
      lastWatched:
      [
        { episode_id: string, content_id: string, position_seconds: number },
        ...
      ]
    }

    Note: same position_seconds carry-over rule as endpoint 14 - only kept when the
    episodeId given here matches the episode that was already tracked for this content.

16) POST /api/profile/:profileId/watch/:contentId/:episodeId/progress
    Purpose: Updates just the in-episode playback position (position_seconds) for an
    episode already tracked in the profile's watch history - meant to be called
    frequently (e.g. every 10-15 seconds during playback, on pause, or before the tab
    closes), unlike endpoints 14/15 above which represent a one-time "started watching
    this" event and reorder the watch history. Does not reorder lastWatched and does not
    write an operation log entry on success, to avoid flooding the logs with a line per
    progress ping.

    req.params =
    {
      profileId: string,
      contentId: string,
      episodeId: string
    }

    req.body =
    {
      position_seconds: number
    }

    res =
    {
      success: boolean,
      message: string,
      positionSeconds?: number
    }

    Note: requires an existing lastWatched entry for this content whose episode matches
    episodeId - call endpoint 14 or 15 first to create it. If no matching entry exists,
    or position_seconds is missing/negative/not a number, this returns success: false
    with a message instead of creating or guessing an entry.

        },
        ...
      ]
    }

18) GET /api/profile/:profileId/top_picks
    Purpose: Suggests up to 3 top-rated content items based on this profile's own taste -
    groups the categories of everything it has watched or liked, takes its top 3 most
    common categories, then returns the highest-rated other content sharing those
    categories (excluding anything already watched or liked).

    req.params =
    {
      profileId: string
    }

    res =
    {
      success: boolean,
      message: string,
      content:
      [
        {
          id: string,
          title: string,
          description: string,
          cover_image_name: string,
          type: "movie" | "series",
          categories: string[],
          release_date: string,
          age_limit: number,

          average_rating: number,

    Note: returns an empty list (not an error) if the profile has no watch/like history yet,
    or if none of its top categories match any other content.

--------------------------------------------------
CONTENT ENDPOINTS (/api/content)
--------------------------------------------------
("Content" means the movies/series in the system. Video files live on episodes, not on content
itself - every watchable item is an Episode, including a movie's single episode.)
(All content endpoints are public - no login needed, and no token is verified anywhere on
this router. Because of this, the age-limit check that exists in the authorization layer
never actually triggers on these routes - it only activates when a profile is attached to
the request first, which never happens here. Age is only enforced on the profile-scoped
routes above (likes, watch, progress) and on review routes below.)

19) GET /api/content/:contentId
    Purpose: Gets the details of one specific movie/series.

    req.params =
    {
      contentId: string
    }

    res =
    {
      success: boolean,
      message: string,
      content:
      {
        id: string,
        title: string,
        description: string,
        cover_image_name: string,
        type: "movie" | "series",
        categories: string[],
        release_date: string,
        age_limit: number,
        likes: number,
        createdAt: string,
        average_rating: number,
        review_count: number,
        imdb_rating: number | null,
        imdb_votes: number | null,
        actors: string[]
      }
    }

    Note: imdb_rating, imdb_votes, and actors are fetched from an external IMDB lookup ONCE,
    at content creation time (POST /api/admin/content), and persisted to the database from
    then on - this endpoint just returns whatever is already stored, with no live lookup on
    every GET. They are null/empty if the lookup failed, found no match, or the IMDB feature
    was disabled on the server at creation time. Every other endpoint that returns a content
    object (search, create, update, suggestions, top picks) includes these same three fields.

        {

          title: string,
          description: string,

    Note: sort accepts any field name and is passed straight to the database with no

--------------------------------------------------
REVIEW ENDPOINTS (/api/reviews)
--------------------------------------------------
(A review is one profile's rating plus an optional comment, for one specific episode.
A profile can have at most one review per episode.)

25) POST /api/reviews/:profileId/:contentId/:episodeId
    Purpose: Adds a review (rating + optional comment) for a specific episode, written by this
    profile. A profile can only review a given episode once - a second attempt returns
    "You have already reviewed this episode". Adding a review automatically recalculates
    that content's average_rating and review_count.

    req.headers =
    {
      Authorization: "Bearer <token>"
    }

    req.params =
    {
      profileId: string,
      contentId: string,
      episodeId: string
    }

    req.body =
    {
      rating: number,
      comment?: string
    }

    res =
    {
      success: boolean,
      message: string,
      review?:
      {
        id: string,
        contentId: string,
        episodeId: string,
        profileId: string,
        rating: number,
        comment: string
      }
    }

    Note: rating must be a valid number, and is validated by the schema (an out-of-range
    value returns "Rating must be between 1 and 10"). Since this route runs through
    profile authorization before content authorization, the age-limit check (if enabled)
    DOES apply here - unlike the public /api/content routes above.

26) PUT /api/reviews/:profileId/:contentId/:episodeId
    Purpose: Edits this profile's own review for a specific episode (rating and/or comment).
    Editing the rating automatically recalculates that content's average_rating.

    req.headers =
    {
      Authorization: "Bearer <token>"
    }

    req.params =
    {
      profileId: string,
      contentId: string,
      episodeId: string
    }

    req.body =
    {
      rating?: number,
      comment?: string
    }

    res =
    {
      success: boolean,
      message: string,
      review?:
      {
        id: string,
        contentId: string,
        episodeId: string,
        profileId: string,
        rating: number,
        comment: string
      }
    }

    Note: fails with "You have not reviewed this episode yet" if no review exists yet for
    this profile/episode pair - this endpoint never creates one.

27) DELETE /api/reviews/:profileId/:contentId/:episodeId
    Purpose: Removes this profile's own review for a specific episode. Removing the review

28) GET /api/reviews/
    Purpose: Searches/browses all reviews using query filters. Public, no login needed.

    req.query =
    {

      episode_id?: string,
      profile_id?: string,

      skip?: number,
      sort?: string,
      sortOrder?: "greater_to_smaller" | "smaller_to_greater"

    Note: filter by episode_id to see all reviews of one episode, or by profile_id (together
    with episode_id) to find one profile's review of one episode. As with content search,
    sort accepts any field name with no whitelist.

--------------------------------------------------
ADMIN ENDPOINTS (/api/admin)
--------------------------------------------------
(Every admin endpoint requires: req.headers = { Authorization: "Bearer <token>" }, and the
logged-in user must have admin permission level, unless noted as super-admin-only.)
(This section consolidates every admin-only action across users, content, profiles, and
reviews.)

29) GET /api/admin/users
    Purpose: Lets an admin search/browse through the list of all users.

    req.query =
    {
      ...search filters supported by the underlying user query builder (e.g. by email,
      phone, full name, birth date range, join date range - see your user model's query
      builder for the exact/current list of supported keys),
      limit?: number,
      skip?: number,
      sort?: "createdAt" | "birthday" | "fullName" | "email",
      sortOrder?: "greater_to_smaller" | "smaller_to_greater"
    }

    res =
    {
      success: boolean,
      message: string,
      users:
      [
        { id: string, email: string, phone: string, fullName: string, birthday: string, createdAt: string, permission_level: number },
        ...
      ]
    }

    Note: unlike content/review search, "sort" here is restricted to a fixed whitelist of
    4 values (createdAt, birthday, fullName, email) - any other value is rejected with an
    error listing the valid options, instead of silently doing nothing or erroring at the
    database level.

33) PUT /api/admin/users/:user_id/permission
    Purpose: Promotes or demotes a user's permission level. An admin cannot assign a
    level higher than their own, and cannot act on a user whose current level is already
    above their own. Acting on a user at the same level as the admin is allowed (e.g. one
    admin can demote another admin back to plain user), as long as the requested new
    level differs from the target's current one. Permission levels: USER = 0, ADMIN = 1,
    SUPER_ADMIN = 2.

    req.params =
    {
      user_id: string
    }

    req.body =
    {
      permission_level: number
    }

    res =
    {
      success: boolean,
      message: string
    }

37) POST /api/admin/users/:user_id/ban
    Purpose: Bans a user for a given number of hours (does not kick existing sessions).

    req.params =
    {
      user_id: string

--------------------------------------------------
GENERAL NOTES
--------------------------------------------------
- Every res always has a "success" field (true or false). Most also have a "message" field,
  except a handful of simple admin actions (tokens_count, kick, ban, ban-check) which only
  return "success" plus their specific data field on success.
- "?" after a field name means it is optional / not always present.
- All admin-only actions live under /api/admin, consolidated by resource type (users,
  profiles, content, reviews) rather than scattered across each resource's own base path.
- req.headers = the request headers (used for the auth token).
- req.params = values that are part of the URL path (like :user_id).
- req.query = values after the "?" in the URL (like ?limit=10).
- req.body = the JSON object sent in the request body.
- sortOrder uses the same two values everywhere: "greater_to_smaller" (descending, default)
  or "smaller_to_greater" (ascending). Any other value returns an error. Note that only the
  admin user-search endpoint restricts which field "sort" can be; content and review search
  pass "sort" straight through with no validation.
- videoUrl lives only on episodes, never on content itself.
- A profile can only have one review per episode - adding a second is rejected, not merged
  or overwritten.
- average_rating (number) and review_count (number) are included directly on every content
  object returned by every endpoint (search, single lookup, admin create/update,
  suggestions, top picks). They stay in sync automatically whenever a review is added,
  edited, or deleted (see endpoints 25-27 and 46-47).
- imdb_rating (number | null), imdb_votes (number | null), and actors (string[]) are also
  included directly on every content object returned by every endpoint. Unlike
  average_rating/review_count, these are set only once, from an external IMDB lookup done at
  creation time (endpoint 39) - they are never re-fetched afterward, so they stay whatever
  they were (or null/empty, if the lookup failed, found no match, or was disabled) for the
  lifetime of that content item.
- Age-limit enforcement is conditional on server configuration AND on whether a profile has
  already been resolved earlier in that route's middleware chain. In practice this means it
  is enforced on profile-scoped content actions (likes, watch, progress, reviews) but NOT on
  the public /api/content browsing routes, since those never resolve a profile at all.
- Every lastWatched entry (endpoints 12, 14, 15) carries a position_seconds field - how
  many seconds into that episode the profile last stopped at. It is only preserved across
  a watch call (14/15) when resuming the exact same episode already tracked for that
  content; switching episodes always resets it to 0. Frequent in-episode updates go through
  the dedicated progress endpoint (16), not the watch endpoints, to avoid reordering the
  watch history or flooding the logs on every playback tick.
